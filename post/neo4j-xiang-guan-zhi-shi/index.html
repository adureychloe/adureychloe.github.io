属性图

  • 关系必须具备类型和方向
  • 原生图数据库和其它数据库的区别在于 无索引邻接(IFA) 的概念。提交数据库事务时，对关系的引用与关系开始和结束处的节点一起存储。
  • 与关系数据库访问相比，IFA的好处是：
    • 更少的索引查找
    • 没有表扫描
    • 减少数据重复
  • 可以将关系模型、NoSQL模型（键值存储、文档存储）实现为图模型。

Cypher

  • 创建关系时如果不指定关系方向，默认是从左到右
  • ON是条件选择语句，类似于“if”
  • 不能单独删除（delete）一个有关系的节点，要用detach delete
  • where p.name <> 'string' 意思是p除了name是string以外的节点
  • 可以用IS NOT NULL来判断存在性
  • 可以用IN来判断列表的包含性
  • 时间类型的值可以用.year，.month等取其中的值
  • 字符类型可以用STARTS WITH，ENDS WITH，CONTAINS来筛选
  • 可以用toLower来小写，toUpper来大写，用其转换字符串属性后，查询引擎将关闭索引index的使用
  • EXPLAIN加在开头来检查是否使用索引index
  • 判断一个pattern在图中是否存在：exists {pattern}，运行效率不如直接查找关系。
  • EXPLAIN和PROFILE的区别：前者提供query steps的估计，不运行语句；后者运行语句，提供精确的query steps和返回行数，能显示一个query的表现
  • 评价query表现的一个指标是DB hits个数

  • 一般返回的节点为json数据，包括labels，identity，properties

  • CASE WHEN THEN ELSE END语句能在RETURN进行条件选择
  • count(*)用来aggregare data，计算返回记录的数量包括null值
  • collect返回list，可以切片，可以用size(list)计算list长度。[]也返回列表。
  • aggregating functions:

  • 模式理解（Pattern comprehension）是在不改变MATCH查询基数的情况下在RETURN中创建列表，行为类似于OPTIONAL MATCH与collect()。格式是：[<pattern> | value], 根据模式将值放入列表中。
  • map函数就是键值对列表 {key:value}
  • duration.between()函数比较两个date或datetime类型的时间数据，返回持续时间值。如果要以天为单位，用duration.inDays().days。
  • apoc用来转换时间格式

Graph Data Science

Intro

GDS 的工作方式是将数据转换并加载到内存格式中，该格式针对高性能图形分析进行了优化。

Workflow

分为三步：

读取和加载图

从数据库中取出数据转换加载到内存，该过程称为 projecting a graph ，内存图称为 graph projection 。GDS能同时有多个图映射，被 Graph Catalog 管理。

执行算法

包括中心度、社区检测、路径搜索、图嵌入等。

存储结果

能够将结果写回数据库，以csv格式导出到磁盘，或将结果流向另一个应用程序或下游工作流程。

并行CPU

社区版最大是4。

内存管理

主要两个：

  • 堆空间：用于存储内存图，执行GDS算法，查询执行和交易状态。

  • 页面缓存：用于索引和缓存存储在磁盘上的Neo4j数据。提高查询数据库和投影图形的性能。

它们的大小是互斥的。

Graph Catalog

用下面命令CALL这个函数：

CALL gds.graph.<command>

列出所有图投影：

CALL gds.graph.list()

从Actor、Movie节点和ACTED_IN 关系创建一个投影：

CALL gds.graph.project('my-graph-projection', ['Actor','Movie'], 'ACTED_IN')

CALL gds.graph.list() YIELD graphName, nodeCount, relationshipCount, schema

导出图

  • gds.graph.expport
  • gds.beta.graph.export.csv

丢弃图

CALL gds.graph.drop('my-graph-projection')

Native Projection

GDS的投影有两种方式：native和cypher。前者针对效率和性能进行了优化，以支持大规模的图形数据科学。Cypher 投影针对灵活性和定制进行了优化，以支持探索性分析、实验和更小的图形投影。

gds.graph.project() 就是本地投影。本地投影需要三个强制性参数：graphName、nodeProjection和relationshipProjection。此外，可选的配置参数允许我们进一步配置图的创建。

Basic Native Projection

CALL gds.graph.project('native-proj',['User', 'Movie'], ['RATED']);

只有一个节点或关系时可以不用列表：

CALL gds.graph.project('native-proj',['User', 'Movie'], 'RATED');

如果要创建的图映射名字已存在，需要把原来的丢弃才能添加。

CALL gds.graph.drop('native-proj');

通配符 ‘*’ 能包含数据库中的所有节点和关系：

CALL gds.graph.project('native-proj','*', '*');

改变关系指向

在 relationshipProjection 中可以设置三种指向选择：

  • NATURAL : 默认，原本的指向
  • REVERSE ：和原来相反的指向
  • UNDIRECTED ：无向图

包含节点和关系的属性

CALL gds.graph.drop('native-proj', false);

CALL gds.graph.project(
    'native-proj',
    ['User', 'Movie'],
    {RATED: {orientation: 'UNDIRECTED'}},
    {
        nodeProperties:{
            revenue: {defaultValue: 0}, // (1)
            budget: {defaultValue: 0},
            runtime: {defaultValue: 0}
        },
        relationshipProperties: ['rating'] // (2)
    }
);

平行关系合并

neo4j中允许多条属性和方向一致的平行关系存在，有时候需要合并这些平行关系。

合并关系不带属性：

CALL gds.graph.project(
  'user-proj',
  ['User'],
  {
    SENT_MONEY_TO: { aggregation: 'SINGLE' }
  }
);

合并关系计算数量：

CALL gds.graph.project(
  'user-proj',
  ['User'],
  {
    SENT_MONEY_TO: {
      properties: {
        numberOfTransactions: {
          // the wildcard '*' is a placeholder, signaling that
          // the value of the relationship property is derived
          // and not based on Neo4j property.
          property: '*',
          aggregation: 'COUNT'
        }
      }
    }
  }
);

Cypher Projections

使用cypher来定义映射模式，更加灵活。虽然Cypher投影提供了更多的灵活性和定制性，但相对于本地投影来说，它们对性能的关注度降低了，因此在大的图形上不会表现得那么快或那么好。

语法

例子：

CALL gds.graph.project.cypher(
  'proj-cypher',
  'MATCH (a:Actor) RETURN id(a) AS id, labels(a) AS labels',
  'MATCH (a1:Actor)-[:ACTED_IN]->(m:Movie)<-[:ACTED_IN]-(a2)
   WHERE m.year >= 1990 AND m.revenue >= 1000000
   RETURN id(a1) AS source , id(a2) AS target, count(*) AS actedWithCount, "ACTED_WITH" AS type'
);

cypher语句可以匹配特定节点和关系。

一旦语句完成，我们就可以像上一课那样应用 degree centrality 方法。只是我们将通过actedWithCount 属性对度中心性进行加权，同时直接将前10名的结果流传回来。这将计算出该演员在最近的高票房电影中与其他演员合作的次数。

CALL gds.degree.stream('proj-cypher',{relationshipWeightProperty: 'actedWithCount'})
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS name, score
ORDER BY score DESC LIMIT 10

什么时候用 Cypher Projections

  • 复杂过滤：需要用到 match/where 的情况。
  • 带权重的多跳路径聚合

过渡到本地投影

//set a node label based on recent release and revenue conditions
MATCH (m:Movie)
WHERE m.year >= 1990 AND m.revenue >= 1000000
SET m:RecentBigMovie;

//native projection with reverse relationships
CALL gds.graph.project('proj-native',
  ['Actor','RecentBigMovie'],
  {
  	ACTED_IN:{type:'ACTED_IN'},
    HAS_ACTOR:{type:'ACTED_IN', orientation: 'REVERSE'}
  }
);

//collapse path utility for relationship aggregation - no weight property
CALL gds.beta.collapsePath.mutate('proj-native',{
    pathTemplates: [['ACTED_IN', 'HAS_ACTOR']],
    allowSelfLoops: false,
    mutateRelationshipType: 'ACTED_WITH'
});

//count actors that acted with the most other actors in recent high grossing movies and stream the top 15
CALL gds.degree.stream('proj-native', {nodeLabels:['Actor'], relationshipTypes: ['ACTED_WITH']})
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS name, score
ORDER BY score DESC LIMIT 15

从cypher到本地投影的流程提示：

  • 使用节点属性条件进行过滤：为符合属性条件的节点添加标签，以便在本地投影中进行过滤。（例子中的RecentBigMovie）

  • 用关系属性条件进行过滤

  • 聚合多跳路径：

    • 看看collapse-path是否能满足你的用例需求。它不对关系进行加权，但结果往往可以与加权的聚合非常相似
    • 对于大投影上的某些类型的问题，可以使用相似性和嵌入算法来近似聚合关系。

Graph Algorithms

Tiers

GDS算法分为三个层级：alpha, beta, production。

  • Production-quality ：算法通过稳定性和扩展性测试，用 gds.来做前缀。
  • Beta: 算法是production-quality层级的候选算法，用 gds.beta. 来做前缀。
  • Alpha：该算法是实验性的，随时可能修改或移除，用 gds.alpha. 来做前缀。

Execution Modes

GDS算法有四种执行模式，用来决定结果怎么处理。

  • stream：将算法的结果作为一个记录流返回。流模式将算法的结果输出到屏幕上，允许你打印结果并直接在控制台中查看。这类似于标准 Cypher 读取查询的操作方式。
  • stats：返回单一记录的汇总统计信息，返回算法计算的统计结果，如计数或百分位数分布，但不写入Neo4j数据库或修改任何数据。
  • mutate：将算法的结果写到内存中的图形投影中，并返回单一记录的汇总统计。
  • write：将算法的结果写回Neo4j数据库，并返回单一记录的汇总统计。

只有production层级的算法才有全部的执行模式。

Memory Estimation

GDS提供一个估算程序来在真正执行前预估在数据上执行算法需要的内存大小。可以在命令后加 .estimate 来执行。

算法结构

CALL gds[.<tier>].<algorithm>.<execution-mode>[.<estimate>](
	graphName: STRING,
	configuration: MAP
)

Centrality and Importance

中心性算法被用来确定图中不同节点的重要性。应用场景：

  • 推荐系统
  • 供应链分析
  • 欺诈和异常性检测

Degree Centrality

度中心性算法是最简单和普遍的中心性算法。它统计一个节点有多少关系。在 GDS 中统计的是出度。

例子：

CALL gds.graph.project('proj', ['Actor','Movie'], 'ACTED_IN');

//get top 5 most prolific actors (those in the most movies)
//using degree centrality which counts number of `ACTED_IN` relationships
CALL gds.degree.stream('proj')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS actorName, score AS numberOfMoviesActedIn
ORDER BY numberOfMoviesActedIn DESCENDING, actorName LIMIT 5

PageRank

PageRank 是一种用于衡量有向图中节点的影响力的算法，特别是在关系意味着某种形式的流动的情况下，如支付网络、供应链和物流、通信、路由以及网站和链接的图。

简单来说，PageRank 通过计算来自邻居节点的传入关系的数量来估计一个节点的重要性，并以这些邻居的重要性和出度中心度加权。基本假设是，更重要的节点可能会有更多来自其他重要节点的传入关系的比例。

下面是一个应用 PageRank 来寻找导演→演员网络中最有影响力的人的例子，这些人来自1990年或以后上映的电影，收入至少为1000万美元。

首先可以使用cypher projection 来得到 (Person)-[:DIRECTED_ACTOR]→(Person) 的图。

//drop last graph projection
CALL gds.graph.drop('proj', false);

//create Cypher projection for network of people directing actors
//filter to recent high grossing movies
CALL gds.graph.project.cypher(
  'proj',
  'MATCH (a:Person) RETURN id(a) AS id, labels(a) AS labels',
  'MATCH (a1:Person)-[:DIRECTED]->(m:Movie)<-[:ACTED_IN]-(a2)
   WHERE m.year >= 1990 AND m.revenue >= 10000000
   RETURN id(a1) AS source , id(a2) AS target, count(*) AS actedWithCount,
    "DIRECTED_ACTOR" AS type'
);

CALL gds.pageRank.stream('proj')
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).name AS personName, score AS influence
ORDER BY influence DESCENDING, personName LIMIT 5

其它度中心性算法

  • Betweenness Centrality：衡量一个节点在图中其他节点之间的程度。它经常被用来寻找作为图中一个部分到另一个部分的桥梁的节点。
  • Eigenvector Centrality：特征向量中心性：衡量节点的横向影响。与PageRank类似，但只对邻接矩阵的最大特征向量起作用，因此不会以同样的方式收敛，并倾向于更强烈地支持高度节点。它在某些用例中可能更合适，特别是那些不定向的关系。
  • Article Rank: PageRank的一个变体，它假定源自低度节点的关系比来自高度节点的关系具有更高的影响力。

Path Finding

路径搜索算法在两个或多个节点之间寻找最短路径，或评估路径的可用性和质量。应用场景：

  • 供应链分析：识别原产地和目的地之间或原材料和成品之间的最快路径
  • 客户旅程：分析构成客户体验的事件。例如，在医疗保健领域，这可能是一个住院病人从入院到出院的经历。

Dijkstra 源-目标最短路径

一个常见的路径寻找算法是Dijkstra。它计算一个源节点和一个目标节点之间的最短路径，支持加权路径。

例子：

CALL gds.graph.project('proj',
    ['Person','Movie'],
    {
        ACTED_IN:{orientation:'UNDIRECTED'},
        DIRECTED:{orientation:'UNDIRECTED'}
    }
);

MATCH (a:Actor)
WHERE a.name IN ['Kevin Bacon', 'Denzel Washington']
WITH collect(id(a)) AS nodeIds
CALL gds.shortestPath.dijkstra.stream('proj', {sourceNode:nodeIds[0], TargetNode:nodeIds[1]})
YIELD sourceNode, targetNode, path
RETURN gds.util.asNode(sourceNode).name AS sourceNodeName,
    gds.util.asNode(targetNode).name AS targetNodeName,
    nodes(path) as path;

其它路径搜索算法

  • 两个节点之间：

    • A* Shortest Path: Dijkstra的一个扩展，使用启发式函数来加快计算速度。
    • Yen’s Algorithm Shortest Path: Dijkstra的一个扩展，允许你找到多条最短路径，即前k条最短路径。

  • 一个源节点和多个目标节点的最短路径：

    • Dijkstra单源最短路径：一个源和多个目标之间最短路径的Dijkstra实现。
    • Delta-Stepping单源最短路径：并行的最短路径计算。计算速度比Dijkstra单源最短路径快，但使用更多内存。

  • 一个源节点和多个目标节点的一般路径：

    • BFS：在每次迭代中，按照与源节点距离增加的顺序搜索路径。
    • DFS：在每次迭代中，沿单一的多跳路径尽可能地搜索。

Community Detection

社区发现算法被用来评估节点组在图中的聚类或分区情况。GDS的大部分社区发现功能都集中在区分和分配这些节点组的ID，用于下游的分析、可视化或其他处理。

应用场景：

  • 欺诈检测：通过识别经常发生可疑交易的账户和/或相互之间共享标识符，找到欺诈团伙。
  • 用户画像
  • 市场细分：根据优先级、行为、兴趣和其他标准，将目标市场划分为可接近的子群体。

Louvain社区发现

一个常见的社区检测算法是 Louvain。Louvain 对每个社区的模块化得分进行最大化，其中模块化量化了将节点分配给社区的质量。这意味着评估一个社区内的节点与随机网络中的节点相比有多大的连接密度。

Louvain 通过一种递归地将社区合并在一起的分层聚类方法来优化这种模块化。有多个参数可用于调整Louvain，以控制其性能和产生的社区的数量和大小。这包括最大的迭代次数和使用的分层级别，以及评估收敛/停止条件的容忍度参数。

另外一个重要的考虑是，Louvain是一个随机的算法。因此，社区分配在重新运行时可能会有一些变化。当图没有一个自然定义好的群落结构时，运行之间的变化会变得更加显著。Louvain包括一个 seedProperty 参数，可以用来分配初始群落ID，并帮助实现运行之间的一致性。另外，如果一致性对你的使用情况很重要，其他的社群检测算法，如弱连通分量（WCC），采取更确定的分区方法来分配社群，因此在运行之间不会改变。

例子：

CALL gds.graph.project('proj', ['Movie', 'Person'], {
    ACTED_IN:{orientation:'UNDIRECTED'},
    DIRECTED:{orientation:'UNDIRECTED'}
});

然后我们可以运行Louvain。在这里，我们将在 mutute 模式下运行Louvain，以保存社区ID，并返回社区数量、分布、模块化得分的高级别统计数据，以及 Louvain 如何处理图形的信息。

CALL gds.louvain.mutate('proj', {mutateProperty:'communityId'})

我们可以用 stream 操作来验证投影中的communityId节点属性。

CALL gds.graph.nodeProperty.stream('proj','communityId', ['Person'])
YIELD nodeId, propertyValue
WITH gds.util.asNode(nodeId) AS n, propertyValue AS communityId
WHERE n:Person
RETURN n.name, communityId LIMIT 10

其它社区发现算法

  • Label Propagation：和 louvain相似，快速算法，并行性好。对于大型图形非常有用。
  • Weakly Connected Components (WCC)：将图划分为连接节点的集合，以便每个节点都可以从同一集合中的任何其他节点到达。不同集合的节点之间不存在路径。
  • Triangle Count:：计算每个节点的三角形的数量。可用于检测社区的凝聚力和图形的稳定性。
  • Local Clustering Coefficient:计算图中每个节点的局部聚类系数，这是节点与其邻居如何聚类的指标。

Node Embedding

节点嵌入的目标是计算节点的低维向量表示，使向量之间的相似性（如点积）接近于原图中节点之间的相似性。这些向量也被称为嵌入，对探索性数据分析、相似性测量和机器学习非常有用。

直觉

下图说明了节点嵌入背后的概念，即在图中靠近的节点最终在二维嵌入空间中靠近。因此，嵌入从图中获取结构，即n维邻接矩阵，并将其近似为每个节点的2维向量。由于维度大大降低，嵌入向量在下游过程中的使用效率更高。例如，它们可以被用于聚类分析，或者作为训练节点分类或链接预测模型的特征。

用例

节点嵌入在多个用例中都有应用，从推荐系统、异常和欺诈检测、实体解析和其他形式的知识图谱补全。

  • 探索性数据分析（EDA）：如在TSNE图中可视化嵌入，以更好地理解图形结构和潜在的节点集群
  • 相似性度量：节点嵌入允许你使用K近邻（KNN）或其他技术来扩展大型图中的相似性推断。
  • 作为机器学习的特征

FastRP

快速随机映射。利用概率抽样技术生成图形的稀疏表示，允许极快地计算嵌入向量，其质量可与传统的随机游走和神经网络技术（如Node2vec和GraphSage）相比。这使得FastRP成为在GDS中开始探索图的嵌入的最佳选择。

微调参数：

  • embeddingDimension :适用于GDS中的所有节点嵌入算法。控制嵌入向量的长度。设置这个参数是对降维和准确性的权衡。更大的嵌入维度将更准确地捕捉图的结构，但也需要更长的时间来生成，并产生需要更多内存和计算来处理下游的嵌入向量。嵌入维度的选择在很大程度上取决于图中节点的数量。由于嵌入所能编码的信息量受到其维度的限制，一个更大的图将倾向于需要一个更大的嵌入维度。一个典型的值是128-1024范围内的2的幂。一个至少为256的值可以在100K节点的图形上得到良好的结果。
  • IterationWeights : 控制两个方面：中间嵌入的迭代次数和影响最终节点的嵌入向量。是一个列表，每一个迭代一个值，代表权重，默认是[0.0, 1.0, 1.0].

FastRP 的一个重要特点是能考虑节点和关系的属性权重。

例子

CALL gds.graph.project('proj', ['Movie', 'Person'], {
    ACTED_IN:{orientation:'UNDIRECTED'},
    DIRECTED:{orientation:'UNDIRECTED'}
});

设置维度为64。

CALL gds.fastRP.stream('proj',  {embeddingDimension:64, randomSeed:7474})
YIELD nodeId, embedding
WITH gds.util.asNode(nodeId) AS n, embedding
WHERE n:Person
RETURN id(n), n.name, embedding LIMIT 10

其它节点嵌入算法

  • Node2Vec：基于随机游走计算向量
  • GraphSage：一种利用节点属性和图结构计算节点嵌入的归纳建模方法。

Similarity

相似性算法用来衡量节点对之间的相似性。在GDS中，这些算法在图的投影上批量运行。当根据用户指定的指标和阈值识别出相似的节点对时，就会在这对节点之间画出一个具有相似性分数属性的关系。根据运行算法时使用的执行模式，这些相似性关系可以被 stream，mutate到内存图中，或写回数据库中。

应用场景：

  • 欺诈检测: 通过分析一组新用户帐户与标记帐户的相似程度来发现潜在的欺诈用户帐户
  • 推荐系统：在一个在线零售商店中，识别与用户当前正在浏览的商品配对的商品，以告知用户印象并提高购买率。
  • 实体解析：根据图中的活动或识别信息，识别出彼此相似的节点

算法

  • Node Similarity：根据图中共享的相邻节点的相对比例来确定节点之间的相似性。在可解释性很重要的情况下，节点相似性是一个很好的选择，你可以把比较的范围缩小到数据的一个子集。缩小范围的例子包括只关注单一社区、新增加的节点，或与感兴趣的子图特定距离内的节点。
  • K-Nearest Neighbor (KNN): 基于节点属性来确定相似性。如果调整得当，GDS KNN的实现可以很好地用于大型图的全局推理。它可以与嵌入和其他图算法一起使用，根据图中的接近度、节点属性、社区结构、重要性/中心性等确定节点之间的相似性。

相似性度量指标

  • Node Similarity 在Jaccard相似度和重叠相似度之间选择。
  • KNN 对度量的选择是由节点的属性类型驱动的。整数列表受 Jaccard和Overlap 的影响，浮点数列表受 Cosine Similarity、Pearson 和 Euclidean 的影响。

控制比较范围

将每个节点与图中的其他节点进行比较是一项计算成本很高的工作，其复杂度大约为 O(n^2)。GDS对节点相似性和KNN的实现都有内部机制，可以智能地选择节点对进行比较，使它们工作得更快，规模更大。

  • Node Similarity 对节点有一个 degreeCutOff 参数，允许你对要选择的节点的度中心性设置一个下限。

  • KNN有各种参数可以调整，以影响节点比较的速度与完整性的权衡，包括采样率、初始采样器方法、迭代间的随机连接数，以及其他一些参数。

控制结果范围

对于相似性比较，我们可能还想控制返回的结果数量，只考虑最相关的节点对。节点相似性和KNN都有一个topK参数来限制每个节点返回的相似性比较的数量。对于节点相似性，也有能力在全局范围内限制结果，而不仅仅是在每个节点的基础上。

KNN例子

CALL gds.graph.project('proj', ['Movie', 'Person'], {
    ACTED_IN:{orientation:'UNDIRECTED'},
    DIRECTED:{orientation:'UNDIRECTED'}
});

FastRP 生成嵌入向量：

CALL gds.fastRP.mutate('proj',  {
    embeddingDimension:64,
    randomSeed:7474,
    mutateProperty:'embedding'
})

使用默认的cosine指标：

CALL gds.knn.stream('proj', {nodeLabels:['Person'], nodeProperties:['embedding'], topK:1})
YIELD  node1, node2, similarity
RETURN gds.util.asNode(node1).name AS actorName1,
    gds.util.asNode(node2).name AS actorName2,
    similarity
LIMIT 10

Graph Machine Learning

使用GDS中的机器学习工具场景：

  • 管理复杂的模型设计：图形数据，由于其高度互联的性质，给机器学习工作流程带来了复杂的问题，对于那些不太熟悉图形的人来说，这些问题可能很难被发现和解决。如果不加以考虑，这些复杂情况会损害ML模型的有效性、计算性能和预测性能。GDS pipeline 考虑这些复杂情况，主要的例子是正确的数据分割设计，处理严重的类不平衡，以及避免特征工程中的数据泄漏。
  • 通过强大的数据库耦合，快速实现生产：因为GDS存在于Neo4j中，所以很容易将GDS ML直接应用于Neo4j数据库。一旦你在GDS中训练了一个管道，该模型就会有效地自动保存和部署--准备好用一个简单的预测命令对Neo4j数据库的数据进行预测。企业用户可以将这些模型持久化，以便重复使用，也可以将它们发布出去，在各个团队之间共享。
  • 开发和实验：即使对于拥有成熟和强大的MLOps工作流程的企业的经验丰富的从业人员来说，原生的ML管道也消除了许多通常与图形机器学习相关的初始摩擦。这使你可以实验和测试模型方法，以快速启动。

GDS的ML管道和模型

GDS专注于为端到端ML工作流程提供管理管道。数据选择、特征工程、数据分割、超参数配置和训练步骤在管道对象中被耦合在一起，以跟踪所需的端到端步骤。

有两种 ML 的管道格式：

  • Node Classification Pipelines：有监督的节点二分类和多分类。
  • Link Prediction Pipelines：对成对的节点之间是否应该存在关系或 "链接 "进行监督预测。

管道都有训练和预测过程。

Node Classification Pipeline

步骤1-6将由管道自动执行。您只需负责为它们提供配置和超参数。

所以整个流程就是：

  1. 投影一个图并配置管道（顺序并不重要）。

  2. 用 train 命令执行管道。

  3. 用 predict 命令进行预测。如果需要的话，预测结果可以通过图形写入操作写回到数据库中。

例子：预测电影类型

设置问题

对于这个例子，我们将训练一个模型来预测图中哪些电影是喜剧，我们将其定义为具有“Comedy”类型的任何电影。 类型目前在图中由节点“Genre”表示。对于这个问题，我们需要将其表示为Movie节点的属性。为了演示目的，我们将分配一个cls属性，如果电影是喜剧，则为1，否则为0。

MATCH(m:Movie)-[:IN_GENRE]->(g)
WITH m , collect(g.name) AS genres
SET m.cls = toInteger('Comedy' IN genres)
RETURN count(m), m.cls;

电影节点也有一些runtime and imdbRating 的属性值丢失，这些特征可能携带预测信息。在现实世界中，我们可能想对这些值进行估算，为了快速演示的目的，我们将通过标签分配将它们从样本中排除。当我们这样做的时候，我们将对电影进行过滤，只考虑那些在2010年或之后发行的电影，因为这种类型的数据可能会随着时间的推移而漂移，使过去的数据变得不太相关。

MATCH(m:Movie)
WHERE m.year >= 2010
    AND m.runtime IS NOT NULL
    AND m.imdbRating IS NOT NULL
SET m:TrainMovie
RETURN count(m)

投影：

CALL gds.graph.project('proj',
    {
        Actor:{},
        TrainMovie:{ properties: ['cls', 'imdbRating', 'runtime']}
    },
    {
        ACTED_IN:{},
        HAD_ACTOR:{type:'ACTED_IN', orientation:'REVERSE'}
    }
);

CALL gds.beta.collapsePath.mutate('proj',
  {
    pathTemplates: [['HAD_ACTOR', 'ACTED_IN']],
    allowSelfLoops: false,
    mutateRelationshipType: 'SHARES_ACTOR_WITH'
  }
) YIELD relationshipsWritten;

配置管道

步骤：

  • 创建管道
  • 添加节点属性
  • 选择节点属性为特征
  • 配置节点拆分
  • 添加候选模型

创建管道：

CALL gds.beta.pipeline.nodeClassification.create('pipe')

然后添加节点属性，一个节点分类管道可以在 mutate 模式下执行一个或几个GDS算法，在投影中创建节点属性。

首先生成嵌入向量,它将封装图中电影节点的局部性：

CALL gds.beta.pipeline.nodeClassification.addNodeProperty('pipe', 'fastRP', {
  embeddingDimension: 32,
  randomSeed: 7474,
  mutateProperty:'embedding'
})
YIELD name, nodePropertySteps;

接下来，我们可以增加 "度中心性"，它将衡量共享演员的其他电影的数量。

CALL gds.beta.pipeline.nodeClassification.addNodeProperty('pipe', 'degree', {
  mutateProperty:'degree'
})
YIELD name, nodePropertySteps;

最后，我们将对运行时间属性进行缩放，这对于像这种与其他属性相比量级相对较高的数值来说是很好的做法。

CALL gds.beta.pipeline.nodeClassification.addNodeProperty('pipe', 'alpha.scaleProperties', {
  nodeProperties: ['runtime'],
    scaler: 'Log',
  mutateProperty:'logRuntime'
})
YIELD name, nodePropertySteps;

一旦配置好了属性，我们就可以配置我们想用作模型特征的节点属性子集了。

CALL gds.beta.pipeline.nodeClassification.selectFeatures(
    'pipe',
    ['imdbRating', 'logRuntime', 'embedding', 'degree'])
YIELD name, featureProperties;

之后，我们可以配置数据分割。在这种情况下，它是相当直接的。我们配置一个testFraction，它决定了如何在测试和训练节点之间随机分割。由于管道使用交叉验证策略，我们也可以在这里设置我们想要的验证 folds 的数量。

CALL gds.beta.pipeline.nodeClassification.configureSplit('pipe', {
 testFraction: 0.2,
  validationFolds: 5
})
YIELD splitConfig;

管道配置的最后一步是创建候选模型。该管道能够以不同的训练方法和超参数配置运行多个模型。训练步骤完成后，将选择性能最好的模型。

为了演示，我们在这里只是添加了几个不同的逻辑回归，有不同的惩罚超参数。

CALL gds.beta.pipeline.nodeClassification.addLogisticRegression('pipe', {penalty: 0.0})
YIELD parameterSpace;

CALL gds.beta.pipeline.nodeClassification.addLogisticRegression('pipe', {penalty: 0.1})
YIELD parameterSpace;

CALL gds.beta.pipeline.nodeClassification.addLogisticRegression('pipe', {penalty: 1.0})
YIELD parameterSpace;

训练管道

训练步骤：

  • 应用节点和关系过滤器
  • 执行上述管道配置步骤
  • 对所有候选模型进行交叉验证训练
  • 根据度量参数选择最佳候选模型。为了便于解释，我们将使用ACCURACY，但也存在其他的F1和精度/召回率指标
  • 在整个训练集上重新训练获胜模型，并根据指标在测试集上进行最终评估
  • 在模型目录中注册获胜的模型
CALL gds.beta.pipeline.nodeClassification.train('proj', {
  pipeline: 'pipe',
  targetNodeLabels: ['TrainMovie'],
  modelName: 'nc-pipeline-model',
  targetProperty: 'cls',
  randomSeed: 7474,
  metrics: ['ACCURACY']
}) YIELD modelInfo
RETURN
  modelInfo.bestParameters AS winningModel,
  modelInfo.metrics.ACCURACY.train.avg AS avgTrainScore,
  modelInfo.metrics.ACCURACY.outerTrain AS outerTrainScore,
  modelInfo.metrics.ACCURACY.test AS testScore;

预测

CALL gds.beta.pipeline.nodeClassification.predict.write(
  graphName: String,
  configuration: Map
)
YIELD
  preProcessingMillis: Integer,
  computeMillis: Integer,
  postProcessingMillis: Integer,
  writeMillis: Integer,
  nodePropertiesWritten: Integer,
  configuration: Map

Link Prediction

GDS目前提供一个二元分类器，目标是0-1的指标，0代表没有链接，1代表有链接。这种类型的链接预测在无向图上效果非常好, 预测单一标签的节点之间的一种关系，比如社交网络和实体解析问题。

在关系拆分中，有一个额外的 feature-input 集在节点属性和特征生成步骤之前。简而言之，这是为了处理数据泄漏问题，即模型特征是使用你试图预测的关系来计算的。这样的情况会让模型在特征中使用通常不会有的信息，导致性能指标过于乐观。

除了数据泄露问题，一般来说，链接预测问题在数据抽样不周全的情况下，会出现严重的类不平衡和性能问题。GDS的实现有多种机制来克服这些问题。总之，它可以归结为抽样和加权程序，以及选择适当的评估指标。

例子：预测演员关系

设置问题

电影推荐数据集就目前而言并不是这种类型的链接预测的最佳候选者，因为它是一个k-partite图，即关系只在不相交的节点集之间进行。在这种情况下，这些集合可以与节点标签保持一致：用户、电影、人物和流派。为了展示一个快速的例子，我们将从图中制造一个社会网络。我们将筛选出高票房的大片，然后在同一部电影的演员之间建立ACTED_WITH关系。这里有几个额外的步骤，以使图真正成为我们需要的无定向图。

//set a node label based on recent release and revenue conditions
MATCH (m:Movie)
WHERE m.year >= 1990 AND m.revenue >= 1000000
SET m:RecentBigMovie;

//native projection with reverse relationships
CALL gds.graph.project('proj',
  ['Actor','RecentBigMovie'],
  {
  	ACTED_IN:{type:'ACTED_IN'},
    HAS_ACTOR:{type:'ACTED_IN', orientation: 'REVERSE'}
  }
);

//collapse path utility for relationship aggregation - no weight property
CALL gds.beta.collapsePath.mutate('proj',{
    pathTemplates: [['ACTED_IN', 'HAS_ACTOR']],
    allowSelfLoops: false,
    mutateRelationshipType: 'ACTED_WITH'
});

//write relationships back to graph
CALL gds.graph.writeRelationship('proj', 'ACTED_WITH');

//drop duplicates
MATCH (a1:Actor)-[s:ACTED_WITH]->(a2)
WHERE id(a1) < id(a2)
DELETE s;

//clean up extra labels
MATCH (m:RecentBigMovie) REMOVE m:RecentBigMovie;

//project the graph
CALL gds.graph.drop('proj');
CALL gds.graph.project('proj', 'Actor', {ACTED_WITH:{orientation: 'UNDIRECTED'}});

这给了我们一个只有演员节点和 ACTED_WITH 关系的图形投影，就像一个 "co-acting "的社会网络。

同样的方法可以用于不同的社交网络推荐问题。例如，如果不是演员相互合作，而是用户彼此是朋友，我们可以使用这样的模型来推荐朋友。同样，在欺诈检测和执法应用程序中，如果我们有彼此认识或互动的嫌疑人和受害者社区，我们可以使用链接预测来推断图中尚未知道的现实世界关系。

配置管道

步骤：

  • 创建管道
  • 添加节点属性
  • 添加链接特征
  • 配置关系拆分
  • 添加候选模型
CALL gds.beta.pipeline.linkPrediction.create('pipe');

然后添加节点属性。在这个例子中，使用 fastRP 节点嵌入，其逻辑是：如果两个演员在ACTED_WITH 网络中彼此接近，他们就更有可能在同一电影中扮演角色。度中心也是另一个潜在的有趣特征，即更多产的演员更有可能与其他演员出现在同一电影中。

CALL gds.beta.pipeline.linkPrediction.addNodeProperty('pipe', 'fastRP', {
    mutateProperty: 'embedding',
    embeddingDimension: 128,
    randomSeed: 7474
}) YIELD nodePropertySteps;

CALL gds.beta.pipeline.linkPrediction.addNodeProperty('pipe', 'degree', {
    mutateProperty: 'degree'
}) YIELD nodePropertySteps;

接下来我们将添加链接特征。这一步配置一个对称函数，从节点对中获取属性并计算链接预测模型的特征。对于这个问题，我们使用余弦距离和L2作为FastRP嵌入，这是对相似性/距离的良好衡量，度中心性的 hadamard 则是衡量2个节点之间总规模的好办法。

CALL gds.beta.pipeline.linkPrediction.addFeature('pipe', 'l2', {
  nodeProperties: ['embedding']
}) YIELD featureSteps;

CALL gds.beta.pipeline.linkPrediction.addFeature('pipe', 'cosine', {
  nodeProperties: ['embedding']
}) YIELD featureSteps;

CALL gds.beta.pipeline.linkPrediction.addFeature('pipe', 'hadamard', {
  nodeProperties: ['degree']
}) YIELD featureSteps;

之后，我们配置关系拆分，设置训练/测试/特征集的比例，负采样比例，以及交叉验证中使用的验证 fold 的数量。对于我们的例子，我们将把关系分成20%的测试、40%的训练和40%的特征输入。这让我们在所有的集合之间有一个很好的平衡。我们还将使用2.0的负向抽样比例。在链接预测的背景下，一个负面的例子是任何节点对之间没有链接。这些是在关系拆分步骤中随机抽样的。

CALL gds.beta.pipeline.linkPrediction.configureSplit('pipe', {
    testFraction: 0.2,
    trainFraction: 0.5,
    negativeSamplingRatio: 2.0
}) YIELD splitConfig;

就像节点分类一样，管道配置的最后一步是创建候选模型。该管道能够以不同的训练方法和超参数配置运行多个模型。训练步骤完成后，将选择性能最好的模型。

为了演示，我们在这里只是添加了几个不同的逻辑回归，有不同的惩罚超参数。

CALL gds.beta.pipeline.linkPrediction.addLogisticRegression('pipe', {
    penalty: 0.001,
    patience: 2
}) YIELD parameterSpace;

CALL gds.beta.pipeline.linkPrediction.addLogisticRegression('pipe', {
    penalty: 1.0,
    patience: 2
}) YIELD parameterSpace;

训练管道

步骤：

  • 应用节点和关系过滤器
  • 执行上述管道配置步骤
  • 对所有候选模型进行交叉验证训练
  • 根据平均精度-召回率（又称AUCPR）选择最佳候选人。
  • 在整个训练集上重新训练获胜的模型，并在测试中用AUCPR做最终评估。
  • 在模型目录中注册获胜的模型。
CALL gds.beta.pipeline.linkPrediction.train('proj', {
    pipeline: 'pipe',
    modelName: 'lp-pipeline-model',
    targetRelationshipType: 'ACTED_WITH',
    randomSeed: 7474 //usually a good idea to set a random seed for reproducibility.
}) YIELD modelInfo
RETURN
modelInfo.bestParameters AS winningModel,
modelInfo.metrics.AUCPR.train.avg AS avgTrainScore,
modelInfo.metrics.AUCPR.outerTrain AS outerTrainScore,
modelInfo.metrics.AUCPR.test AS testScore

预测

CALL gds.beta.pipeline.linkPrediction.predict.stream('proj', {
  modelName: 'lp-pipeline-model',
  sampleRate:0.1,
  topK:1,
  randomSeed: 7474,
  concurrency: 1
})
 YIELD node1, node2, probability
 RETURN gds.util.asNode(node1).name AS actor1, gds.util.asNode(node2).name AS actor2, probability
 ORDER BY probability DESC, actor1
