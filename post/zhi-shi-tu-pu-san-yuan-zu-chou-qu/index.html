可以看到它主要是利用语法结构来进行信息提取。我们要抽三元组用到的就是其中的语义角色标注SRL（Semantic Role Labeling）。

语义角色标注SRL

自然语言分析技术大致分为三个层面：词法分析、句法分析和语义分析。语义角色标注是实现浅层语义分析的一种方式。

在一个句子中，谓词是对主语的陈述或说明，指出“做什么”、“是什么”或“怎么样，代表了一个事件的核心，跟谓词搭配的名词称为论元。语义角色是指论元在动词所指事件中担任的角色。例如下面：

A0奥巴马是施事者，发表是谓词（predicate），A1演说是受事者。

具体来说，语义角色标注的任务就是以句子的谓词为中心，研究句子中各成分与谓词之间的关系，并且用语义角色来描述他们之间的关系，即句子的谓词（Predicate）- 论元（Argument）结构。

抽取三元组的话一般把A0和A1作为实体，谓词predicate作为关系。

# 使用语义角色标注构造关系抽取
def srl_AtoA(sent):
    rst = ltp.pipeline([sent],tasks=["cws","pos","srl"]) # cws是分词，pos是词性标注，srl是语义角色标注
    seg = rst.cws[0]
    srl = rst.srl[0]
    results = []
    for s in srl:
        key = s['predicate'] # 谓词
        values = s['arguments'] # 论元
        start = ''
        end = ''
        for value in values:
            # 语义角色只需要获取A0,A1(施事和受事）
            if value[0] == 'A0':
                start = value[1]
            if value[0] == 'A1':
                end = value[1]
        if start != '' and end != '':
            results.append((start,key,end))
    return results

问题很多，一个一个来说。

1、A0和A1并不一定是单独实体

基于语义角色标注得到的三元组很多并不是我们想要的命名实体，而是一段文字。

2、实体和关系没有分类

实体没有分类，这两个“Entity1”和“Entity2”还是我前面在插入之前自己定义的，实际上如果不定义也能插入，就所有节点都是一类：

但是实体应该分类，因为我们要抽取的实体肯定是有不同类别的，就像项目书里写的：

所以我们肯定是得事先定义好要抽取的实体类型。

关系也一样，如果单纯将抽出来的谓词当作关系，那基本上关系类型都是独立的，也不能体现不同类型实体之间固定关系（比如某地属于某国这种固定关系）。

3、实体都是孤立的

可以看到17对实体全部是孤立的，没有相互联系。一方面是因为只有一篇文章，但更重要是因为抽取效果差，比如水族馆位于德国柏林这个信息都没有抽取出来。说明这种方法不适合。

官网示例

我们来看一下neo4j官网的示例。一般来说输入是csv表，我们需要两个实体表，一个关系表。

在插入数据之前，为了确保主键的值是唯一的，需要定义数据表约束：

CREATE CONSTRAINT UniqueProduct ON (p:Product) ASSERT p.id IS UNIQUE;
CREATE CONSTRAINT UniqueOrder ON (o:Order) ASSERT o.id IS UNIQUE;

这里Product和Order就是我们现在定义的两种节点类型，也就是实体类型，id是插入每个数据时自动分配的索引。

一个实体表是products.csv：

它包含商品ID，商品名字，单价，这些都是product实体的属性。

另一个实体表是order.csv:

它包含订单ID，订单日期，国家，这些都是order实体的属性。

关系表是order-details.csv:

它包含商品ID、订单ID和订单数量。其中商品ID和订单ID的对应代表了两个实体的关系，该关系拥有属性订单数量。

我们要定义的图结构是这样：

CONTAINS就是我们后面定义的关系类别，代表了product和order两个实体之间的关系。

分别导入实体和关系数据：

  • 导入product数据：
LOAD CSV FROM 'file:///products.csv' AS row
WITH toInteger(row[0]) AS productId, row[1] AS productName, toFloat(row[2]) AS unitCost
MERGE (p:Product {productId: productId})
  SET p.productName = productName, p.unitCost = unitCost
RETURN count(p);

​

  • 导入order数据：
LOAD CSV WITH HEADERS FROM 'file:///orders.csv' AS row
WITH toInteger(row.orderID) AS orderId, datetime(replace(row.orderDate,' ','T')) AS orderDate, row.shipCountry AS country
MERGE (o:Order {orderId: orderId})
  SET o.orderDateTime = orderDate, o.shipCountry = country
RETURN count(o);
  • 导入关系：
LOAD CSV WITH HEADERS FROM 'file:///order-details.csv' AS row
WITH toInteger(row.productID) AS productId, toInteger(row.orderID) AS orderId, toInteger(row.quantity) AS quantityOrdered
MATCH (p:Product {productId: productId})
MATCH (o:Order {orderId: orderId})
MERGE (o)-[rel:CONTAINS {quantityOrdered: quantityOrdered}]->(p)
RETURN count(rel);

这里面的CONTAINS就是我们定义的关系类型。

设计知识图谱

由上可知，我们需要事先定义好知识图谱的类别和关系，才能构建好一个知识图谱，方便我们后续的查询等操作。举例来说可以设计这样的图谱：

  • 创建“人物”实体
  • 创建“组织”实体
  • 创建“产品”实体
  • 创建“领域”实体
  • 创建“型号”实体
  • 创建“人”和“人”的关系，这个关系的属性可能有亲属、同事等
  • 创建“人”和“组织”的关系，属性可能有职位等
  • 。。。。。

这只是可能的一个例子，具体设计什么实体、关系和属性要根据具体的数据和任务来决定。

实体关系抽取

要抽取实体关系，我们得首先定义两个东西：（1）什么是我们需要的实体；（2）什么是我们需要的实体关系。一般来说，可以用schema来描述有价值的实体关系类型。

通用信息抽取

根据语义结构直接抽三元组行不通，我又尝试了一下开放域的通用信息抽取方法 工具。

UIE

中科院软件所和百度共同提出了一个大一统诸多任务的通用信息抽取技术 UIE（Unified Structure Generation for Universal Information Extraction），发表在ACL‘22。UIE在实体、关系、事件和情感等4个信息抽取任务、13个数据集的全监督、低资源和少样本设置下，UIE均取得了SOTA性能，不需要标注数据（或仅需少量标注数据），即可快速完成各类信息抽取任务。

直接用的话准确率可以，查全率不太行。

实体识别：
